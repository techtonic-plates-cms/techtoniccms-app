# ABAC API Migration & UX Overhaul Plan

## Executive Summary

This plan migrates the TechtonicCMS frontend from the **old flat rule schema** to the **new typed value schema** and re-designs the entire ABAC GUI for **inexperienced users** who do not know what "ABAC", "attribute path", or "policy priority" mean.

**Design principle:** _Everything should be explainable in one sentence to someone who has never heard of access control._ Use plain language, guided flows, templates, and visual feedback. Hide complexity behind progressive disclosure. Never expose raw IDs. Always explain consequences.

The work is split into four phases. Phases 1–2 are prerequisites for correctness; Phases 3–4 deliver the beginner-friendly UX.

- Run `deno task codegen:gql` against the current API endpoint (requires `DEV_API_URL`).
- Verify the generated types contain the new `PolicyRule.value` union and updated `CreatePolicyInput` / `UpdatePolicyInput` shapes.

### 1.2 Update type definitions in `policies.remote.ts`

- Replace `PolicyRule.expectedValue: string` and `PolicyRule.valueType: string` with the new `value` discriminated union.
- Update `Policy` type if any new fields were added (e.g. `description` on rules already exists).

### 1.3 Update GraphQL documents in remote functions

- `POLICIES_QUERY`, `POLICY_QUERY`: replace `expectedValue valueType` with the new `value { stringValue numberValue booleanValue uuidValue dateTimeValue arrayValue contextReferencePath }` selection set.

- `addPolicyRule` mutation: build the `value` object from form inputs instead of passing `expectedValue` + `valueType`.

- `src/routes/(app)/settings/policies/[id]/+page.svelte`

## Phase 2: Policy Rule Builder — Beginner-First Design

The current rule builder exposes `valueType` and `expectedValue` as raw text inputs. A beginner does not know what a "value type" is or which one to pick. The new design must **infer everything possible** and only ask for what the user actually knows.

### 2.1 Create a reusable `PolicyRuleBuilder` component

**Location:** `src/lib/components/policy-rule-builder.svelte`

**Beginner-first behavior:**

- **No `valueType` dropdown.** The component infers the correct `value` field from the operator and the attribute selected.
- **Plain-language labels** for every attribute:
  - `SUBJECT_ROLE` → "User's role"
  - `RESOURCE_ENTRY_CREATED_BY` → "Who created this entry"
  - `ENVIRONMENT_CURRENT_TIME` → "Current time"
- **Operator dropdown in plain English:**
  - `Eq` → "is exactly"
  - `Ne` → "is not"
  - `In` → "is one of"
  - `Gt` → "is greater than"
  - `EqContextRef` → "is the same as" (with context: "is the same as the current user")
  - `IsNull` → "is empty / not set"
  - etc.
- **Dynamic value input** — the user never sees the word "value type":
  | Operator | What the user sees |
  |----------|-------------------|
  | `Eq`, `Ne` | A single input whose type depends on the attribute (text, number, toggle, date picker, or a searchable collection picker for UUIDs) |
  | `Gt`, `Gte`, `Lt`, `Lte` | Number input or date-time picker |
  | `In`, `NotIn` | A tag input / multi-select (e.g. pick multiple roles from a list) |
  | `Contains`, `StartsWith`, `EndsWith`, `Regex` | Plain text input with a small hint below (e.g. "Enter text to match") |
  | `IsNull`, `IsNotNull` | **Nothing** — a sentence reads: "This rule matches when [attribute] is empty" |
  | `EqContextRef` | A dropdown of contextual comparisons: "the current user", "the current time", etc. |
- **Rule connector** shown as a sentence:
  - `AND` → "All of these rules must match"
  - `OR` → "Any one of these rules is enough"
- **Inline examples:** Every rule row shows a live plain-English sentence of what it does:
  - _"Allow if the user's role is one of: editor, admin"_
  - _"Allow if the entry was created by the current user"_
- **Priority slider with presets:** Instead of a raw number, offer:
  - "Standard (100)" — normal allow rules
  - "Creator override (500)" — creator permissions
  - "Admin override (1000)" — admin-level rules
  - "Block / Deny (2000)" — deny rules that override everything
  - Advanced users can still enter a custom number.

### 2.2 Policy Templates — "I want to..." wizard

**Location:** `src/routes/(app)/settings/policies/create/+page.svelte`

Before showing the raw rule builder, offer **one-click templates** based on the most common scenarios from the manual:

| Template                                   | What it creates                                                                |
| ------------------------------------------ | ------------------------------------------------------------------------------ |
| "Let someone view content"                 | Allow Read on Entries + Collections, no rules                                  |
| "Let someone edit their own content"       | Allow Update on Entries, rule: `ResourceEntryCreatedBy EqContextRef SubjectId` |
| "Let someone manage a specific collection" | Allow all actions on Entries, rule: `ResourceEntryCollectionId Eq {uuid}`      |
| "Make an editor"                           | 3 policies: Create, Read, Update-own on Entries                                |
| "Make a viewer"                            | Read-only policies for Entries, Collections, Assets                            |
| "Block access after hours"                 | Deny Read on Entries, rules: time < 9am OR time > 5pm, priority 2000           |
| "Custom rule"                              | Opens the full rule builder                                                    |

Each template pre-fills the form. The user can tweak before saving. This turns 15+ minutes of policy building into 2 clicks.

### 2.3 Replace rule builder on policy create page

- Show template gallery first. "Custom rule" opens the full builder.
- Integrate `<PolicyRuleBuilder bind:rules />` for custom mode.
- Serialize rules to JSON for the hidden `rulesJson` field, using the new `value` shape.
- **Live summary:** Below the form, always show a plain-English sentence of what this policy does:
  - _"This policy will ALLOW reading Entries when: the user's role is 'editor' AND the entry was created by the current user."_

### 2.4 Replace rule builder on policy detail page

- Show each rule as a plain-English sentence instead of raw fields:
  - _"User's role is one of: editor, admin"_ instead of `SUBJECT_ROLE In ["editor","admin"]`
- Replace the "Add Rule" inline form with a compact inline `<PolicyRuleBuilder>`.

### 2.5 Add "Duplicate Policy" button

- **Location:** `src/routes/(app)/settings/policies/[id]/+page.svelte`
- New remote function `duplicatePolicy` that copies the policy with name `"Copy of {name}"`.
- Show a toast/confirmation: "Policy copied. You can now edit it."

### 2.6 Contextual help tooltips

- Every field gets a `?` tooltip with a one-sentence explanation:
  - **Priority:** "Higher numbers are checked first. Deny rules should usually be higher than Allow rules."
  - **Effect:** "Allow = grant access. Deny = block access, even if other rules say yes."
  - **Rule connector:** "AND means every rule must be true. OR means any rule can be true."

## Phase 3: Assignment UX — No Raw IDs, Ever

The current UI forces admins to copy-paste raw UUIDs into text inputs. A beginner does not know what a UUID is, let alone where to find one. Every assignment must use **searchable pickers** with human-readable names.

### 3.1 Create reusable picker components

All pickers follow the same pattern: type a name, see matching results, click to select. No IDs visible.

**`src/lib/components/role-picker.svelte`**

- Searchable combobox. Shows role name + description.
- Emits `roleId` on select. The parent never sees the ID.

**`src/lib/components/policy-picker.svelte`**

- Searchable combobox. Shows: name, effect badge (Allow/Deny), resource, action.
- Grouped by effect for quick scanning.
- Emits `policyId` on select.

**`src/lib/components/user-picker.svelte`**

- Searchable combobox. Shows: name, status badge (Active/Inactive).
- Emits `userId` on select.

### 3.2 User detail page — `/settings/users/[id]`

Redesign the entire page for clarity:

- **Roles section:**
  - Current roles shown as cards/tags with "Remove" buttons.
  - "Add role" opens `<RolePicker />` + optional **expiration date** picker.
  - Expired roles shown greyed out with a "Expired on {date}" label.
- **Policies section (currently missing):**
  - Same pattern as roles: show assigned policies as cards with effect badges.
  - "Add policy" opens `<PolicyPicker />` + optional expiration.
- **Status actions with confirmation:**
  - Deactivate: "This will log the user out everywhere immediately. Continue?"
  - Ban: "This will prevent the user from logging in. Continue?"
- **"View Effective Permissions"** button (Phase 4.2) — a beginner-friendly way to understand what this user can actually do.

### 3.3 Role detail page — `/settings/roles/[id]`

- **Policies section:** Show assigned policies as a list with effect badges, resource, action.
- "Add policy" opens `<PolicyPicker />` + optional expiration date.
- **Members section:** Show users with their status. Click a user to go to their detail page.

### 3.4 Policy detail page — `/settings/policies/[id]`

- **Assignments section:** Show which roles and users have this policy, with expiration dates.
- "Assign to a role" opens `<RolePicker />`.
- "Assign to a user" opens `<UserPicker />`.
- Both support optional expiration.

### 3.5 Update remote functions to support `expiresAt`

- `assignRole` → accept optional `expiresAt` string.
- `assignPolicyToRole` → accept optional `expiresAt` string.
- `assignPolicyToUser` → accept optional `expiresAt` string.

## Phase 4: Debugging & Understanding — "Why was this denied?"

Beginners get confused when something is denied. These pages turn debugging from a mystery into a clear answer.

### 4.1 Audit Log page — "What happened?"

**New route:** `/settings/audit`
**New remote function:** `src/lib/remotes/audit.remote.ts`

- **Plain-language table:** Instead of raw GraphQL fields, show:
  - "When" — timestamp
  - "Who" — user name (not ID)
  - "Tried to" — "read entries" instead of `ENTRIES:READ`
  - "Result" — a green checkmark for Allowed, red X for Denied
  - "Why" — the `decisionReason` in large text, e.g.:
    - "Allowed because the policy 'Editor can read' matched"
    - "Denied because the policy 'No after-hours access' blocked it"
    - "Denied because no policy allows this"
- **Pre-filtered views:** Quick links: "Show only denied requests", "Show only my checks".
- Add to sidebar under Settings as "Activity Log" (not "Audit" — less intimidating).

### 4.2 Permission Simulator — "Can this user do this?"

**New route:** `/settings/permissions/check` (or a modal on user detail)
**New remote function:** `src/lib/remotes/permissions.remote.ts`

A beginner-friendly form:

- "Pick a user" — `<UserPicker />`
- "What do they want to do?" — Plain language dropdown:
  - "View content" → `READ` on `ENTRIES`
  - "Edit content" → `UPDATE` on `ENTRIES`
  - "Upload a file" → `UPLOAD` on `ASSETS`
  - etc.
- "Which content?" — Optional collection/entry picker
- **Result shown as a big card:**
  - Green: "Yes — allowed by policy 'Editor can read entries'"
  - Red: "No — blocked by policy 'No after-hours access'"
  - Grey: "No — no policy gives this user permission"
- Explain _why_ in plain English, referencing policy names the user can click to inspect.

### 4.3 Policy "Who does this affect?" preview

- Before saving a new/edited policy, show: "This would affect **3 users** (Alice, Bob, Charlie) and **2 roles** (Editor, Admin)."
- If backend cannot compute this, show a simpler estimate: "This applies to all users with the selected roles."
- This helps beginners understand the blast radius before clicking Save.

## Key Schema Differences (Old → New)

| Aspect                 | Old (Current Code)                            | New (Per Manual)                                                                                                                   |
| ---------------------- | --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| Rule value             | `expectedValue: string` + `valueType: string` | `value: { stringValue? \| numberValue? \| booleanValue? \| uuidValue? \| dateTimeValue? \| arrayValue? \| contextReferencePath? }` |
| Operators              | 14 shown, missing `EQ_CONTEXT_REF` in UI      | 15 operators, `EQ_CONTEXT_REF` uses `contextReferencePath`                                                                         |
| Null checks            | `expectedValue: ""` + `valueType: "STRING"`   | `IsNull`/`IsNotNull` **omit `value` entirely**                                                                                     |
| Array checks           | `expectedValue: "a,b,c"` (fragile)            | `value: { arrayValue: ["a", "b", "c"] }`                                                                                           |
| Context ref            | Not supported in UI                           | `EqContextRef` → `value: { contextReferencePath: SubjectId }`                                                                      |
| Assignment expirations | Not supported in UI                           | `expiresAt: DateTime` on role/user/policy assignments                                                                              |

## Design Principles for Inexperienced Users

These principles govern every decision in this plan:

1. **Never expose raw IDs.** UUIDs are implementation details. Users see names, avatars, and badges.
2. **Use plain English, not jargon.** `SUBJECT_ROLE` → "User's role". `EqContextRef` → "is the same as".
3. **Prefer templates over blank forms.** A beginner should be able to create a working policy in 2 clicks.
4. **Show, don't tell.** Live plain-English summaries of what a policy does. Visual badges for Allow/Deny.
5. **Progressive disclosure.** Simple defaults first; advanced options hidden behind "Show advanced".
6. **Explain consequences.** Every dangerous action (deactivate, ban, delete) explains what will happen.
7. **Help is always one click away.** Tooltips (`?` icons) on every field with one-sentence explanations.
8. **Use familiar patterns.** Role assignment works like adding someone to a group chat. Policy assignment works like adding an app to a folder.
9. **Visual feedback for everything.** Green = allowed. Red = denied. Grey = inactive/expired. No ambiguity.
10. **No dead ends.** If a list is empty, show a friendly message and a "Get started" button, not a blank table.
