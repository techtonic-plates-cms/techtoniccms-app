# Activity Log (Audit) Implementation Plan

## Goal

Replace the placeholder `/settings/audit` page with a fully functional activity log that lists permission-check audit entries from the GraphQL API, plus a detail view for individual audit records.

## Context

- The GraphQL API now exposes `AuditQuery` with `audits(...)` (connection) and `audit(id: UUID!)` (single).
- Generated GraphQL types in `techtonic-client-gql` have **not** been refreshed yet — the new `AuditQuery`, `AuditsConnection`, `AbacAuditFilterInput`, and `AbacAuditSortInput` types are missing.
- Existing list pages (users, policies, api-keys) follow a consistent pattern: remote function → `.svelte` page with metrics cards, filters, table, pagination.
- The `(app)` layout already handles auth via `requireAuth()`, so audit pages do not need to call it again.

---

## Step 1: Regenerate GraphQL Types

Run the codegen to pull the new audit schema into TypeScript types. This makes type-checking reliable for the rest of the work.

```bash
DEV_API_URL=<api_url> deno task codegen:gql
```

> If `DEV_API_URL` is not set in the environment, supply it inline or export it first.

## Step 2: Create `src/lib/remotes/audit.remote.ts`

Add a remote file with two queries following the raw-string + manual-typing pattern used by `users.remote.ts` and `policies.remote.ts`.

### Queries to implement

1. **`getAudits`** — paginated list
   - Accept `first`, `after`, `where`, `order`.
   - Return `{ nodes: Audit[], pageInfo: PageInfo }`.
   - Query shape (assumed from API definition and sibling patterns):
     ```graphql
     query Audits(
     	$first: Int
     	$after: String
     	$where: AbacAuditFilterInput
     	$order: [AbacAuditSortInput!]
     ) {
     	audit {
     		audits(first: $first, after: $after, where: $where, order: $order) {
     			nodes {
     				id
     				requestedAction
     				resourceType
     				decision
     				decisionReason
     				evaluationTimeMs
     				timestamp
     				ipAddress
     				userAgent
     				sessionId
     				user {
     					id
     					name
     				}
     				evaluatedPolicyIds
     				matchingPolicyIds
     			}
     			pageInfo {
     				hasNextPage
     				endCursor
     			}
     		}
     	}
     }
     ```

2. **`getAudit`** — single record for detail view
   - Accept `id: string`.
   - Return `Audit | null`.
   - Same field selection as above.

### Filter building (inside `getAudits`)

Build the `where` object dynamically from URL params:

| Param             | Filter construction                                                                |
| ----------------- | ---------------------------------------------------------------------------------- |
| `decision`        | `{ decision: { eq: value } }`                                                      |
| `resourceType`    | `{ resourceType: { eq: value } }`                                                  |
| `requestedAction` | `{ requestedAction: { eq: value } }`                                               |
| `search`          | `{ user: { name: { contains: value } } }` (if supported by `AbacAuditFilterInput`) |
| `after`           | Passed as `after` cursor                                                           |

Default `order`: `{ timestamp: DESC }` (newest first).

## Step 3: Redesign List Page (`src/routes/(app)/settings/audit/+page.svelte`)

Replace the placeholder with a full list page matching the users/policies layout.

### Structure

1. **Header**
   - Title: "Activity Log"
   - Subtitle: "Permission check history — who tried to do what and why"
   - Remove the back arrow (this is a top-level settings page, not a sub-page).

2. **Metrics cards** (4-column grid, same pattern as users/policies)
   - **Total Checks** — `nodes.length`
   - **Allowed** — count of `decision === 'ALLOW'`
   - **Denied** — count of `decision === 'DENY'`
   - **Avg Eval Time** — average `evaluationTimeMs` + "ms"
   - Use copper left-border for Allowed, destructive for Denied.

3. **Filters** (horizontal row, same pattern)
   - Search input: "Search by user..." (filters by user name if API supports it; otherwise client-side fallback).
   - Decision select: All, Allow, Deny.
   - Resource type select: All + `BaseResource` enum values.
   - Action type select: All + `PermissionAction` enum values.
   - "Clear filters" button when any filter is active.
   - Changing a filter resets `after` cursor (goes to page 1).

4. **Table** (`rounded-lg border`, same table style)
   - Columns:
     - **Time** — `timestamp` formatted as locale string + relative time (e.g., "2 min ago"). Use JetBrains Mono for the raw timestamp.
     - **User** — `user.name`, linked to `/settings/users/{user.id}`.
     - **Action** — plain English: "{requestedAction} on {resourceType}" (e.g., `READ on USERS`).
     - **Decision** — Badge: `ALLOW` → moss-green/copper variant, `DENY` → destructive variant.
     - **Reason** — Truncated `decisionReason` (max ~60 chars), with full text shown on hover via `title` attribute.
     - **Eval Time** — `{evaluationTimeMs}ms` in mono font.
   - Each row is clickable, linking to `/settings/audit/{id}`.
   - Row hover: `hover:bg-muted/50`.

5. **Empty state**
   - ClipboardList icon in a muted circle.
   - "No activity found"
   - If filters active: "Try adjusting your filters"
   - If no filters: "Activity will appear here when permission checks are performed"

6. **Pagination**
   - "Load more" button when `pageInfo.hasNextPage`.
   - Preserve active filters in the `?after=` URL.

### Styling notes

- Follow `DESIGN.md`: no shadows, earthy palette, flat surfaces, Inter for UI, JetBrains Mono for timestamps/IDs.
- Badges should read as "stains" — low-opacity backgrounds with matching text.

## Step 4: Create Detail Page (`src/routes/(app)/settings/audit/[id]/+page.svelte`)

A read-only detail view for a single audit entry. This keeps the list clean while surfacing technical details (IP, UA, policy IDs) that don't belong in a dense table.

### Structure

1. **Header**
   - Back arrow linking to `/settings/audit` (preserves query params if possible, or just `/settings/audit`).
   - Title: "Audit Entry"
   - Subtitle: timestamp in full ISO + relative time.

2. **Summary card**
   - Decision badge (large).
   - Sentence: "**{user.name}** attempted **{requestedAction}** on **{resourceType}** and was **{allowed/denied}**."
   - Evaluation time: "Evaluated in {evaluationTimeMs}ms"

3. **Details sections**
   - **Decision Reason** — full text in a card (this can be long).
   - **Request Context** — IP address, User Agent, Session ID (each shown as a labeled row; omit if null).
   - **Policies Evaluated** — list of `evaluatedPolicyIds` as links to `/settings/policies/{id}`.
   - **Policies Matched** — list of `matchingPolicyIds` as links to `/settings/policies/{id}`.
   - **User** — link to `/settings/users/{user.id}` with user name.

4. **Not-found state**
   - If `getAudit` returns null: "Audit entry not found." with a back link.

## Open Questions / Decisions

1. **Codegen dependency**: Do we want to require running `codegen:gql` first, or should the audit remote use entirely manual types (like the existing remotes) to avoid blocking on an API URL?  
   _Recommendation_: run codegen if `DEV_API_URL` is available; otherwise write manual types and run codegen later.

2. **User name filtering**: The exact shape of `AbacAuditFilterInput` is unknown until codegen runs. If it does **not** support `user: { name: { contains } }`, should we:  
   a) Omit user search from filters, or  
   b) Fetch all and filter client-side?  
   _Recommendation_: implement the filter optimistically; if the API rejects it, fall back to client-side filtering or remove it.

3. **Detail page necessity**: Is a `[id]` detail page needed now, or should the list show all data inline?  
   _Recommendation_: include the detail page — `decisionReason`, `ipAddress`, `userAgent`, `evaluatedPolicyIds`, and `matchingPolicyIds` are too verbose for a table row.
